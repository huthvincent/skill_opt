## 第一层：背景——"skill / 经验库"是什么

想象一个 AI 客服 agent。它每次接待用户，除了靠模型本身的能力，还靠一段**说明书**——里面写着"遇到退款先查订单状态""用户报错代码 X 时先让他重启"这类经验条目。这段说明书就是我们说的**经验库（skill）**。

三个关键点：
1. 说明书写得好，agent 办事漂亮；写得差，处处碰壁；
2. 说明书是**纯文字**、随时可增删改——不需要重新训练模型；
3. **我们的研究问题：怎么自动把这段说明书越改越好？**

我们不动 AI 模型本身（模型冻结、调 API），只优化文字。这叫 **training-free（免训练）**：便宜、快、换模型即用。

## 第二层：别人怎么做的，以及共同盲区

目前最强的方法（SkillOpt、Training-Free GRPO、ACE、GEPA）套路一致：

```
让 agent 带着当前说明书做一批练习题 → 看对错 → AI 反思"为什么错"
→ 改说明书 → 再做题验证 → 循环
```

有效，但有个共同前提：**练习题必须是有标准答案的干净考试题，且必须现场重新做**。这带来三个盲区：

1. 依赖 benchmark 标准题（现实里没有）；
2. 依赖自动判分（现实里用户不会告诉你"答对了没"）；
3. 只能用"当前说明书现场做的题"——**历史上真实用户留下的海量聊天记录，它们一条都用不上**。

## 第三层：我们的切入点（motivation，从未变过）

现实里最多的数据恰恰是那些用不上的：**真实用户和 AI 的交互日志**。它们的特点：
- 是过去的旧版 agent 产生的（术语：off-policy）；
- 没有标准答案，只有模糊的**隐式信号**（用户重新问了一遍？不耐烦了？直接走了？）；
- 一次性的，不能"重做一遍"。

**我们的论文要回答：有这些日志时，怎么把 skill 优化做得比"只会做练习题"的方法更好。** 这个 motivation 从立项到现在没变过一个字。

## 第四层：第一版方法为什么失败了（$18 买来的教训）

第一版的做法：从日志里挖出失败模式 → 让强模型**一口气写 12 条**经验 → **整库塞进** agent 的说明书 → 看整体成功率。

结果：基线 70%，塞了库反而趋势向下。起初以为是实验事故（确实也有事故），但把三个独立证据放在一起，指向的是同一个结论：

| 证据来源 | 内容 |
|---|---|
| 我们的实验 | 12 条整库注入，未见收益、疑似有害 |
| Skill-RM 论文的消融实验 | 把知识裸堆进 prompt：**−2.9 分**；结构化使用：+2.3 分 |
| SkillOpt 论文最扎眼的数字 | 它最终生效的说明书**只由 1–4 条"通过验证的编辑"构成**，且实验证明"限制编辑数量"打败"随便改写" |

**通俗地说：给员工的桌上贴 12 张便利贴，他反而不会干活了。** 机理有三个：
1. **稀释注意力**：条目多了，真正关键的那条被淹没；
2. **误触发**：针对 A 场景的建议在 B 场景被错误执行；
3. **过度谨慎**：一堆"注意这注意那"让 agent 畏手畏脚、对话超时。

所以失败毙掉的不是"日志→skill"这条路，而是"**批量生成、整库倾倒**"这个姿势——而这个姿势恰恰是所有能 work 的方法（SkillOpt）明确避免的。

## 第五层：重设计 v1——"一次只买一条能力"

新方法可以用**医生开药**来理解，六步：

### ① 诊断哪里最痛（日志挖掘，不变）
从日志里找出 agent 老翻车的地方，聚成"失败簇"，按三个标准排序：**发生频率**（常见病优先）× **可修复性**（AI 判断能不能靠一条经验治好）× **是否过时**（用"replay 探针"检查：让现在的 agent 回到日志里的岔路口，看它还犯不犯这个错——不犯了就说明这条日志过时，扔掉）。

### ② 一次只开一种药（单簇提案）
只处理最痛的那一个失败簇，写 **3 个候选条目变体**（同一个病的三种处方），每条附带"**触发签名**"——描述这条经验适用于什么情境。

### ③ 小范围试药（定向微验证——重设计的心脏）
每个候选条目做**带 vs 不带的对照实验**，但**不在全部任务上测，只在这个失败簇对应的 8–12 个任务上测**，另加 5 个"本来就做得对"的任务当**无害对照**（确保新药不把没病的治出病）。

### ④ 严进宽出的准入门
只有同时满足两条才准进说明书：靶点任务上的提升**把测量误差算进去后依然为正**（置信下界 > 0，不是"看起来涨了就行"）；无害对照不掉分。没过的候选进"落选档案"，喂给 AI 当反面教材。

### ⑤ 按症状服药（条件注入）
准入的条目**不是永远挂在说明书里**——只在当前对话匹配它的触发签名时才注入。针对退款问题的经验，不会出现在改密码的对话里。这是对"便利贴淹没效应"的第二重保险。

### ⑥ 处方上限（库生命周期）
说明书最多 5 条；越往后准入越严（编辑预算衰减）；每条持续记录"带它/不带它"的胜率，跌破线就退役。

## 第六层：为什么这次应该能 work（最重要的一段）

**统计上的道理，用一个比喻**：想知道感冒药有没有效，你给 10 个感冒病人吃，效果一目了然；你给全村 1000 人吃再看"全村平均健康分"，什么也看不出来。

第一版就是"给全村吃药看平均分"：我们实测发现，全集 60 次测试的成功率天然波动 ±10 个百分点——任何真实的小提升都会被噪声淹没。重设计改成"给病人吃药"：失败簇任务上基线失败率 50% 以上，一条有效经验能带来 30–50 个百分点的提升，**8–12 个任务就能测得清清楚楚**。

**这还让"日志"这个卖点变得更硬了**：SkillOpt 要花大钱现场做题才能发现哪里错；我们的日志**免费**告诉我们哪里错、错得多不多——省下来的预算全部花在"逐条验证"上。论文术语叫"**日志提供验证经济学**"：知道病灶在哪，试药就又便宜又准。

**逻辑上的保底**：就算 12 条候选里只有 2 条真有用，这套流程会精确地放进那 2 条、挡住其余 10 条——整体只会变好不会变坏（无害对照兜底）。这是"批量倾倒"在结构上做不到的。

## 第七层：跟别人的对比（更新版）

| | 失败从哪发现 | 验证怎么做 | 准入判据 | 注入方式 | 能用真实日志? |
|---|---|---|---|---|---|
| **SkillOpt** | 昂贵的现场做题探索 | 全局验证集（对稀有错误功效低） | 严格提升（噪声下失效） | 单一全局文档 | ❌ |
| **TF-GRPO** | 同题反复做，比较好坏 | 无逐条验证 | 无准入门 | 整库注入 | ❌ |
| **ACE / GEPA** | 现场跑题自我反思 | 自己判自己 | 弱 | 单一 playbook/prompt | ❌ |
| **ExpeL 等离线派** | 自己带标签的旧练习 | 无验证 | 无 | 整库注入 | ❌（非真实 session） |
| **我们 v1** | **日志免费给出失败簇** | **簇匹配定向小样本 + 无害对照** | **置信下界 + 双信号** | **触发签名条件注入** | ✅ 核心卖点 |

三句话记住差异：①别人花钱探索哪里错，我们从日志免费读出哪里错；②别人在全集上做低功效验证，我们在靶点上做高功效验证；③别人整库倾倒，我们逐条准入、按需注入。

## 第八层：Sanity Check v2——花小钱回答"这套到底行不行"

按"最快证伪"排序的四道门，**任何一道红灯当场停**（完整版见 `sanity_check.md`）：

### G0 可控性预门（~$2）——"agent 教得动吗？"
挑 3 个 agent 稳定失败的任务，让强模型**看着标准答案**写"金条目"（允许作弊到底），看 agent 能不能翻案。
**为什么先做它**：第一版失败还有一个没排除的备择解释——也许 haiku 这个小模型根本不听说明书的话。如果连完美指令都执行不了，后面全是空谈。**红灯处理**：换 sonnet 当 agent 重测；再不行才质疑路线。

### G1 单条目闭环（~$5）——"方法的最小完整流程能产生真收益吗？"
完整走一遍新方法：日志聚簇 → 3 个候选变体 → 定向验证 → 准入门。**只要有 1 个变体通过准入门，方法闭环就成立。**
这等于把第一版测砸的"天花板测试"用正确姿势重做——不是问"12 条塞进去涨不涨"，而是问"1 条经过验证的条目在靶点上是否有真实提升且全局无害"。

### G2 信号质量（~$4）——"没有标准答案时，我们判断'哪些日志是失败'判得准吗？"
用手上已有的 ~170 条带隐藏答案的轨迹（第一版实验的遗产，零生成成本）：让 AI 裁判在看不到答案的情况下打分，对照隐藏答案算准确度（AUC ≥ 0.75 过线）。这是"从日志挖失败"那半的地基。

### G3 同预算对垒（~$40）——"核心主张：打得过纯 on-policy 吗？"
给两边同样的做题预算：一边纯现场探索（SkillOpt 姿势），一边用日志定向（我们）。谁准入的条目更多更好、或同等成果省 1/3 以上预算，谁赢。**这是论文成败之门，前三门全过才跑。**

**费用总账**：G0+G1+G2 ≈ **$11** 就能探明重设计成不成立；全部四门含余量 ~$76。全程 haiku、每个脚本带自动熔断、烧钱前先报备。

**手里的家底全部复用**：70% 基线（60 集）直接当"日志"用、12 个已知失败任务就是现成失败簇、上次生成的 12 条库当候选起点、170 条带答案轨迹当 G2 评测集——第一版烧的 $18 一分不浪费。

## 第九层：论文的三个卖点（写 intro 的骨架）

1. **新设定**：第一个把真实交互日志（off-policy、无标准答案、隐式反馈）用进 skill 优化的框架，附"藏起标准答案"的可测量实验协议；
2. **新方法——验证经济学**：日志失败簇 → 定向小样本验证（小 n 大效应）→ 噪声校准准入门 → 条件注入；
3. **新发现**：用实测数据展示"批量注入为什么有害"（我们自己的失败数据 + 机理分析），以及"只信 AI 裁判优化会跑偏多少、双信号锚能拉回多少"。
