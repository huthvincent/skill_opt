# 我们的方法讲清楚 + 和别人的方法对比

> 面向非代码读者。目标：读完你能一句话跟别人解释"我们在做什么、比别人强在哪"。
> 技术正典在 `../REPLAY/PROPOSAL.md`，本文是通俗版。

---

## 一、先说清楚"skill / 经验库"是什么

想象一个 AI 客服 agent。它每次接待用户，靠的是背后的一段**说明书**——里面写着"遇到退款先查订单状态""用户报错代码 X 时先让他重启"这类经验条目。这段说明书就是我们说的**经验库（skill）**。

- 说明书写得好，agent 就办事漂亮；写得差，就到处碰壁。
- 这段说明书是**纯文字**、可以随时增删改的（不是去重新训练模型）。
- **我们的研究问题：怎么自动把这段说明书越改越好？**

关键点：我们**不动 AI 模型本身**（模型是冻结的、调 API 的），只优化这段文字说明书。这叫 **training-free（免训练）**——便宜、快、换个模型就能用。

---

## 二、现有方法怎么做的（以及共同的盲区）

目前最强的几个方法（SkillOpt、Training-Free GRPO、ACE、GEPA）套路都类似：

```
让 agent 带着当前说明书去做一批"练习题" → 看哪些做对哪些做错
→ 让另一个 AI 反思"为什么错" → 改写说明书 → 再做一批练习题验证 → 循环
```

这套路很有效，但有三个**共同盲区**，全都源于"练习题从哪来、对错怎么判"：

1. **练习题必须是干净的标准题**：它们全部依赖有标准答案的 benchmark 题目（像考试题，对错分明）。
2. **对错必须能自动判定**：靠的是"标准答案比对"或"程序检查"（比如数据库状态对不对）。
3. **只能用"现在这版说明书"现场做的题**：每次都要 agent 用当前说明书重新做一遍题（术语：on-policy 在线采样）。历史上用别的说明书、或者真实用户留下的记录，它们**用不上**。

**但现实里最多的数据恰恰是它们用不上的那种**——真实用户和 AI 的海量聊天记录：
- 没有标准答案（用户不会告诉你"你刚才答对了没"）；
- 只有**模糊的隐式信号**（用户是不是重新问了一遍？是不是不耐烦了？是不是直接走了？）；
- 是**过去用别的 agent 产生的**（术语：off-policy 离线数据），没法拿来"重做一遍"。

这就是我们的切入点。

## 三、我们的方法：REPLAY（一句话 + 展开）

**一句话**：让**便宜但脏**的真实用户日志负责"**提出**该怎么改说明书"，让**贵但准**的现场练习负责"**裁决**这个改动到底行不行"。

这个名字来自强化学习里的 "experience replay（经验回放）"——我们把它搬到了纯文字的世界。展开成三步：

### 第 1 步：日志提出候选（便宜的一半）
翻大量真实用户日志，用一个 AI 裁判 + 隐式信号找出"agent 在哪类情况下老翻车"，针对每类翻车**提出**一条候选经验条目（"以后遇到这种情况应该……"）。
> **立场：日志提出的改动只是"嫌疑人"，绝不直接写进说明书。**

### 第 2 步：判断这条日志还算不算数（我们最独特的部件）
真实日志是"过去别的 agent"留下的，可能早就过时了。所以对每条日志证据，我们让**现在的 agent** 回到那个当时的岔路口，看它现在还会不会犯同样的错：
- 还会犯 → 这条经验有用，权重高；
- 已经不会了 → 这条日志过时，降权/丢弃。

> 这是强化学习里 "off-policy 校正（重要性加权）"的**文字版**——据我们调研，还没人在文字空间做过这件事，是方法论上最出彩的一点。

### 第 3 步：花小钱现场验证，宁可保守也不乱改（贵的一半）
把候选条目拿到真实环境里做**对照实验**（带这条 vs 不带这条，同样的题），只有真的变好才采纳。而且判定用**悲观门控**：
- 我们不像别人那样"分数涨一点就采纳"——因为 AI 裁判本身有噪声、还会被"讨好式回答"骗到。
- 我们用多个裁判 + 扰动，取**置信下界**：只有"就算把裁判的误差都算进去还是变好了"才采纳。
- 额外要求：这条改动不能只是讨好裁判，还得在真实日志的隐式信号上也不掉分。

**为什么要"日志提出 + 现场裁决"两条腿？** 因为真实日志便宜、量大、贴近真实用户，但脏、不可信；现场练习准、可信，但贵。单靠日志会被噪声带偏，单靠现场练习又贵又脱离真实用户。我们让便宜的那半负责"广撒网找线索"，贵的那半只花在"核实最值得核实的线索"上——省钱又更准。

（附带第三个小部件 **条目级归因**：说明书里有几十条经验，出了问题到底该改哪条？我们让 agent 记录每次用了哪几条，长期统计每条的"带它/不带它"胜率，精准定位该动哪一条，而不是把整本说明书推倒重来。）

## 四、和现有方法逐一对比

| 方法 | 优化什么 | 练习题从哪来 | 对错怎么判 | 能用真实用户日志吗 | 能防"被裁判骗"吗 |
|---|---|---|---|---|---|
| **SkillOpt**（微软, 2026-05） | 一份 SKILL.md 文档 | benchmark 标准题 | **只用**标准答案/程序检查 | ❌ | ❌（严格提升门，遇噪声就崩） |
| **Training-Free GRPO**（腾讯, 2025-10） | 经验库条目 | benchmark 标准题，同题反复做 | 主要靠标准答案 | ❌ | ❌ |
| **ACE**（2025-10） | 一份不断长大的"playbook" | agent 自己跑的题 | agent 自己判 | ❌ | ❌（自己判自己，易自我确认） |
| **GEPA**（2025-07） | 提示词（prompt） | 现场跑的题 | 任务分数 | ❌ | ❌ |
| **ExpeL / AutoGuide**（离线派） | 经验/规则 | 过去的题，但是**agent 自己带标签的**练习 | 需要成功标签 | ❌（不是真实用户 session） | ❌ |
| **Skill-RM**（2026-06） | —（把裁判本身做成 skill，**没有优化环节**） | — | — | ❌ | 它就是个裁判，不优化 skill |
| **我们 REPLAY** | 经验库条目 | **真实用户日志（提出）+ 现场练习（裁决）** | AI 裁判 + 隐式信号 + 现场对照 | ✅ **这是核心卖点** | ✅ **悲观门控 + 真实信号锚定** |

**三句话记住差异**：
1. 别人的练习题都是"标准考试题"，我们能吃"真实用户的乱聊记录"。
2. 别人假设裁判可信，我们假设裁判有噪声、会被骗，专门设计了防线。
3. 别人只会"现在这版说明书现场重做题"，我们能把过去的旧记录也榨出价值（还带了个判断旧记录过没过期的机制）。

## 五、为什么这是一篇能发的论文（不是小修小补）

我们调研了 38 篇相关工作，确认了一个**没人填的空白**：

> 没有任何方法能同时用"**便宜有偏的真实离线日志**"和"**贵但可信的现场练习**"来优化一份文字经验库。两拨人各做各的——做日志挖掘的不产出可复用 skill，做 skill 优化的不碰真实日志。

三个可写进论文的贡献：
1. **新设定**：第一个从真实交互日志优化经验库的框架，还配了一个"藏起标准答案"的可测量实验协议（能精确量出每个部件值多少分）。
2. **新方法**：日志过期判断（replay 探针）+ 只核实最可疑线索（分歧路由）+ 防被裁判骗（悲观门控）。
3. **新发现**：量化"只信 AI 裁判去优化"会把说明书带偏多少，并证明真实日志信号能把它拉回来。

**唯一要盯紧的竞争对手**：Skills-Coach（2026-04）已经把"Training-Free GRPO + 合成练习题"这个组合占了。所以我们的论文**必须死守"真实日志"这个卖点**——这也正是我们和所有人的根本区别。
