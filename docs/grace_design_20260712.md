# GRACE：接地准入的自我进化 harness（正结果导向，2026-07-12）

> 目标：不做负结果，做一个能出正结果、且打赢现有 self-evolve agent 的系统。
> 核心转向：**停止"从对话重建真值"（结构性难，验证≈生成），改为"把 harness 建在信号已接地处，贡献做在'怎么把自我进化做对'"。**
> 工作名 GRACE = Grounded-admission Continual Evolution。

## 一、所有实验逼出来的硬约束（任何方案必须遵守）
1. **好库真能帮**：oracle 库 +9.4pp（Sonnet）。媒介 work。✓
2. **文字裁判把关 → 库净零**：+0.0pp，因为裁判可骗、放行毒条目。✗
3. **杀手失败模式**：现实假成功 68/70 是"做错了动作"（错件/错款），不是"没做"。
4. **区分对/错动作需要知道"正确答案"**——纯从对话重建它，和解任务一样难（J4=0.63）。
5. **但"给定正确 spec，检查执行"很容易**：J4_oracle=0.90。→ **难点不在检查，在拿到接地信号。**
6. **Line-5 铁律**：自我进化能 work 的全骑执行级可验证信号；文字裁判全败。
7. **博客**：评估器是头号瓶颈、应置于循环外；reward hacking 普遍。

**结论**：不要造一个"重建真值的裁判"。要**把准入信号锚在真实可得的执行接地信号上**，让自我进化建在被证明有效的信号属性上。

## 二、GRACE 设计：准入靠"接地"，不靠"评判"

### 2.1 两层接地信号（优雅退化 = 泛化贡献）
- **Tier-1（可验证域：AppWorld / 代码 / 工具-执行）**：准入信号 = **环境自带的执行验证器**（单元测试 / DB 状态检查）。这正是 ACE +17% 骑的信号，在这些域**优化时合法可用**（不是作弊，环境本就提供）。→ 拿**头条正结果**。
- **Tier-2（部分信号域：tau2）**：无完整 gold 验证器时，准入 = **意图接地 + 显式约束核查**（见 2.3）+ 配对 A/B。→ 证明"接地准入"在信号更弱处**仍能救回大部分增益**——这是超越 ACE（只能在有执行信号处 work）的泛化故事。

### 2.2 库的单元：自带证据的条目（provenance-at-write-time）
不存漂浮的"建议"（可骗裁判放行），存**每条都引用支撑它的已验证执行**的条目（博客 Chain-of-Evidence）："在情境 P 下，动作序列 A 在任务 T 上被验证器确认达成了用户请求"。**条目的有用性由构造保证，不由裁判印象保证。**

### 2.3 核心新机制：约束核查 > spec 重建（这是绕开 J4 那堵墙的关键）
J4 卡住是因为**重建完整 spec** 太难。但抓 68 条错动作假成功，不需要完整 spec——只需**抽取用户显式陈述的约束**（少、高精度）核对最终状态：
- "用户点名订单 W123 → agent 改的是 W456 → 违约"；
- "用户要换红色 → DB 里换成了绿色 → 违约"。
**抽取用户显式约束**（高精度、易抽）比**重建正确终态**（低精度、难）容易得多——这是把 Tier-2 从 0.63 往 0.85 推的具体杠杆，也是可发的方法论点。

### 2.4 自我进化循环
第三方蒸馏候选（非自我总结）→ 接地核查（Tier-1 验证器 / Tier-2 约束核查）→ **配对反事实 A/B**（加此条目能否在匹配任务上提升**接地**成功率）→ 统计门（LCB/FDR）→ 准入。红队硬化防"过拟合验证器/特定任务"（Goodhart）。QD 保留防坍缩。

### 2.5 成功判据（正结果目标，硬指标）
**held-out 上的接地/真值成功率**打赢：无库 baseline、文字裁判把关版（=我们 CAGE-v1 失败模式）、ACE、SkillOpt、GRASP。**AppWorld 先拿正结果，tau2 用 Tier-2 补泛化。**

## 三、为什么这条大概率出正结果
1. **骑在被证明 work 的信号属性上**（Line-5 铁律）——不再和"重建真值"这堵墙硬碰。
2. **oracle 增益证明上限存在**：好库 +9.4pp 已知；接地准入约等于把 oracle 库的质量筛出来，理应拿到真实正增益。
3. **失败早已定位为纯准入质量问题**（+9.4 vs +0.0 的差距 100% 在准入）——换接地准入，直接对症。

## 四、为什么打得赢 baseline（新颖性，防"这不就是 ACE"）
1. **接地 vs 非接地准入的干净消融**（科学核心）：同一 harness，准入换文字裁判 → 退回 +0.0（CAGE-v1）；换接地 → 正增益。这个对照 ACE 从没做过。
2. **优雅退化到部分信号域**（Tier-2 约束核查）：ACE 只能在有执行信号处 work；GRACE 在 tau2 这种无完整验证器处用约束核查救回大部分——泛化贡献。
3. **对抗硬化 + 溯源 + 防坍缩**：ACE 无红队、无 provenance、无 anti-Goodhart；证明 GRACE 在优化压力下不退化而 ACE 会。

## 五、第一个便宜的"正结果"实验（Exp-P1，~$100-200）
**AppWorld 上：接地准入自我进化 vs (a) 无库 (b) 文字裁判把关版。**
- 假设：接地准入版在 held-out 验证成功率上显著 > 两者。
- 若成立 → **正结果存在**，按 §二建完整 GRACE + tau2 Tier-2 + baseline 对齐。
- 若不成立 → 说明连接地准入都推不动 AppWorld（与 ACE +17% 矛盾，需查实现）——但这概率低，因为 ACE 已证明该信号能驱动增益。
**这是朝正结果打的第一锤，不是又一个负结果判定。**

## 六、诚实风险
- **novelty 拷问**："和 ACE 差别"必须靠 §四三点立住；单看 Tier-1 会像 ACE 增量，Tier-2 约束核查 + 接地消融是真差异化，别只做 Tier-1。
- **Tier-2 会不会也够不着**：约束核查比 spec 重建易，但能不能到 0.85 仍需 Exp-0.5 式小测（可与 Exp-P1 并行，几美元）。
- **AppWorld 工程量**：比 tau2 重（457 API/9 app），需预留搭环境时间。
- **别过度建**：先用 Exp-P1 拿到"接地准入 > 文字准入"这一个正信号，再逐步加红队/QD，不要一次性建满。
