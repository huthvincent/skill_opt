## 二、决定性的误差分解（那台 server 自己做的，很聪明）
J4 = **Stage1 从"用户请求+政策"推 spec** → **Stage2 观察 DB 最终状态** → **Stage3 比对**。
把 Stage1 换成 gold 完美 spec（J4_oracle）→ AUROC **0.626 → 0.903**。
→ **接地的"观察状态+比对"机器是好的（0.90 天花板）；整堵墙 100% 在 Stage1「从自然语言推出正确 spec」。**

## 三、为什么朴素 J1 会失败，J4 为什么是对的方向（核心 insight）
根因：**70 条假成功里 68 条真的执行了一个类型匹配的成功写调用**——现实的假成功不是"没做"，而是"**做错了**"（错订单/错范围/错值）。
- **J1 核对的是 agent 自己的声称**（"我换货了"→有没有换货调用→有）→ 只抓得住"伪造确认号"（18 条），抓不住"执行错动作"（28 条两裁判皆骗）。
- **J4 核对的是用户请求**（"把蓝衬衫换成红的"→DB 里换的是不是那件）→ **能抓"做错了"**，所以假阳率从 0.71 砍到 0.34。
- **方法论金句**：接地要对着**用户意图**（请求+政策，部署可得），不是对着 **agent 的自利声称**。J1（声称接地）catch 伪造，J4（意图接地）catch 错动作——后者才是多数。这是一个干净、可发的方法论点，且已有数据支撑（J1 假阳率 0.60 vs J4 0.34）。

## 四、真实结论：不是"接地是死路"，是"墙搬到了一个具体、便宜、无 gold 的子问题"
- ❌ 旧解读（summary 的保守版）："接地无法区分对/错动作，需要 gold，死路，停。"
- ✅ 修正解读（oracle 分解证明的）："接地机器好使（0.90 天花板）；J4 用**纯部署可得信息**已把假阳率砍半、AUROC 到 0.63；剩下的全部难度 = **Stage1 从用户请求+政策推出接近 gold 质量的 spec**——这是个 well-defined、无 gold、几美元可测的子问题。"

**诚实的悬而未决点**：Stage1（从意图推正确 spec）会不会秘密地和"解任务本身一样难"（verification≈generation）？论据两面——利好：裁判只需知道**目标**（oracle 证明知道目标就够），不需规划执行路径，且可用更强模型+显式政策+慢慢读；风险：读懂"换哪件、按什么政策"本身要真 reading+政策推理，一次性 Sonnet 只到 0.626。**能不能工程到 0.85 是真开放问题——但便宜可测。**

- **AUROC 冲过 ~0.85** → 意图接地成立，**现在才建 CAGE-v2**（判别器用意图接地，不是声称接地），且建在被证明有效的信号上；
